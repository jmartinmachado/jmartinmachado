  Cada skill vive en .claude/skills/<nombre>/ y tiene esta estructura:

  .claude/skills/git-branch-new/
  ├── SKILL.md           ← instrucciones + frontmatter (obligatorio)
  ├── references/        ← docs auxiliares (schemas, ejemplos)
  │   └── examples.md
  └── scripts/           ← helpers ejecutables (bash, python)
      └── slugify.sh

  El SKILL.md tiene 3 secciones que forman el contrato:

  ┌──────────────────────────────────────────────────────────────────┐
  │ 1. FRONTMATTER  (YAML)            ← obligatorio                  │
  │    name + description                                            │
  │    lo unico que Claude usa para decidir CUANDO                   │
  │    invocar el skill (matchea description vs prompt)              │
  │                                                                  │
  │ 2. STEPS  (## Steps)              ← el COMO                      │
  │    procedimiento numerado paso a paso                            │
  │    lo que Claude ejecuta una vez activado                        │
  │    puede incluir bash, scripts/, lecturas de references/         │
  │                                                                  │
  │ 3. RULES  (## Rules)              ← guardrails                   │
  │    constraints duros: "nunca X", "siempre validar Y"             │
  │    previenen que Claude se desvie o rompa cosas                  │
  │    ej: "Never branch from main unless hotfix"                    │
  └──────────────────────────────────────────────────────────────────┘

  Tecnicamente solo el frontmatter es obligatorio para que el skill
  cargue. Sin Steps no hace nada util. Sin Rules Claude puede
  improvisar de mas.

FASE 1 — PLANIFICACION · obtener tarea + crear rama
  ┌──────────────────────────────────────────────────────────────────┐
  │ /git-branch-new PROJ-123                                         │
  │                                                                  │
  │ 1. lee el detalle de la tarea en tickets/backlog.md              │
  │    - ID:           PROJ-123                                      │
  │    - titulo:       "DELETE /items/{id} con guard de uso"         │
  │    - epic / descripcion / referencias cruzadas                   │
  │                                                                  │
  │ 2. infiere tipo de branch desde el ID + naming correcto:         │
  │    PROJ-XXX        → PROJ-XXX-short-slug          (feature)      │
  │    PROJ-FIX-XXX    → PROJ-FIX-XXX-short-slug      (bug fix)      │
  │    PROJ-REL-XXX    → release/PROJ-REL-XXX         (release)      │
  │    hotfix/...      → hotfix/PROJ-FIX-XXX-slug     (urgente)      │
  │                                                                  │
  │ 3. detecta el base branch correcto                               │
  │    - feature              → develop                              │
  │    - release              → develop                              │
  │    - hotfix               → main                                 │

  │                                                                  │
  │ 4. chequea duplicados de ticket                                  │

  │    - no existe     → procede                                     │
  │                                                                  │
  │ 5. stash automatico si hay cambios sin commitear                 │
  │    git stash push -m "auto-stash before branch"                  │
  │                                                                  │
  │ 6. checkout base + pull + crea rama                              │
  │    git checkout develop && git pull                              │
  │    git checkout -b PROJ-123-delete-item-endpoint                 │
  │                                                                  │

FASE 4 — CORRECCION Y VALIDACION · loop iterativo
  ┌──────────────────────────────────────────────────────────────────┐
  │ /git-ship-review   (orquestador)                                 │
  │                                                                  │
  │ STEP 1  - git-commit                                             │
  │   verifica branch (develop / main → ABORT)                       │
  │   genera conventional commit:                                    │
  │     feat(items): add DELETE endpoint                             │
  │     fix(parser): handle empty payload                            │
  │   stagea + commitea + pushea                                     │
  │                                                                  │
  │ STEP 2  - ci-test-report   (validacion automatica)               │
  │   corre toda la suite de tests del proyecto:                     │
  │     go test ./...  → unit + integration tests                    │
  │     parsea coverage por archivo                                  │
  │     detecta tests rotos / nuevos / sin cubrir                    │
  │   ¿pasaron los tests?                                            │
  │     NO → para acá, fixeas el codigo, vuelve al STEP 1            │
  │     SI → genera tabla markdown con coverage                      │
  │                                                                  │
  │ STEP 3  - git-pr-create                                          │
  │   detecta base branch + sync con remote                          │
  │   abre PR con Summary + tabla de Coverage + Test plan            │
  │                                                                  │
  │ STEP 4  - git-pr-review   (validacion semantica)                 │
  │   analiza archivo por archivo + impacto cruzado:                 │
  │     ●  BLOCKER     no se puede mergear                           │
  │     ▲  WARNING     revisar antes                                 │
  │     ○  SUGGESTION  mejora opcional                               │
  │                                                                  │
  │ ¿hay BLOCKERs?                                                   │
  │   SI  → fixeas en local → vuelve al STEP 1 (loop)                │
  │   NO  → PR listo para mergear a develop                          │
  └──────────────────────────────────────────────────────────────────┘

  Validacion en 2 capas:
    ─ ci-test-report  → tests verifican que el codigo funciona
    ─ git-pr-review   → analisis semantico verifica que esta bien hecho
  El loop se repite hasta que las dos capas devuelven OK.
FASE 5 — RELEASE · RC + validacion
  ┌──────────────────────────────────────────────────────────────────┐
  │ /git-release-cut                                                 │
  │                                                                  │
  │ 1. crea rama release/v1.16.0 desde develop                       │
  │                                                                  │
  │ 2. genera Release Candidate                                      │
  │      git tag v1.16.0-rc1                                         │
  │      go build → bin/v1.16.0-rc1/                                 │
  │      bin/current → rc1                                           │
  │                                                                  │
  │ 3. validacion del RC                                             │
  │      - smoke tests sobre el binario                              │
  │      - pruebas manuales sobre endpoints criticos                 │
  │      - verificacion de migraciones de DB                         │
  │                                                                  │
  │ ¿pruebas ok?                                                     │
  │   NO → fix en release branch → nuevo RC (rc2, rc3...)            │
  │   SI → merge release → main                                      │
  │                                                                  │
  │ 4. publicacion final                                             │
  │      git tag v1.16.0  (sin sufijo rc)                            │
  │      go build → bin/v1.16.0/api/app                              │
  │      bin/current → v1.16.0  (symlink atomico)                    │
  │                                                                  │
  │ El proceso de despliegue siempre apunta a bin/current.           │
  │ Nunca se hardcodea la version en ningun lado.                    │
  └──────────────────────────────────────────────────────────────────┘
