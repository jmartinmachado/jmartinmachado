SYS> SKILL ANATOMY
  Each skill lives in .claude/skills/<name>/ with this structure:

  .claude/skills/git-branch-new/
  ├── SKILL.md           ← instructions + frontmatter (required)
  ├── references/        ← auxiliary docs (schemas, examples)
  │   └── examples.md
  └── scripts/           ← executable helpers (bash, python)
      └── slugify.sh

  SKILL.md has 3 sections that form the contract:

  ┌──────────────────────────────────────────────────────────────────┐
  │ 1. FRONTMATTER  (YAML)            ← required                    │
  │    name + description                                            │
  │    the only thing Claude uses to decide WHEN                     │
  │    to invoke the skill (matches description vs prompt)           │
  │                                                                  │
  │ 2. STEPS  (## Steps)              ← the HOW                      │
  │    numbered step-by-step procedure                               │
  │    what Claude executes once activated                           │
  │    can include bash, scripts/, reads from references/            │
  │                                                                  │
  │ 3. RULES  (## Rules)              ← guardrails                   │
  │    hard constraints: "never X", "always validate Y"              │
  │    prevent Claude from going off track or breaking things        │
  │    eg: "Never branch from main unless hotfix"                    │
  └──────────────────────────────────────────────────────────────────┘

  Technically only the frontmatter is required for the skill to load.
  Without Steps it does nothing useful. Without Rules Claude can
  improvise too much.

  Real example — SKILL.md of git-branch-new:

  ┌──────────────────────────────────────────────────────────────────┐
  │ ---                                                              │
  │ name: git-branch-new                                             │
  │ description: >                                                   │
  │   Use when the user wants to create a new git branch             │
  │   Activate when the user says "crea una rama"                    │
  │   or passes a ticket ID like "PROJ-014"                          │
  │ ---                                                              │
  │                                                                  │
  │ # New Branch                                                     │
  │ Creates a branch from the correct base following                 │
  │ the project naming conventions.                                  │
  │                                                                  │
  │ ## Steps                                                         │
  │ 1. Timestamp                                                     │
  │ 2. Extract ticket ID and description                             │
  │ 3. Derive branch name                                            │
  │ 4. Check for duplicate ticket branches                           │
  │ 5. Detect base branch                                            │
  │ 6. Handle dirty working tree (auto-stash)                        │
  │ 7. Checkout base and pull                                        │
  │ 8. Create the branch                                             │
  │ 9. Confirm with structured output                                │
  │                                                                  │
  │ ## Rules                                                         │
  │ - Never branch from main unless explicitly hotfix                │
  │ - Never push the branch in this skill                            │
  │ - Keep description slug short (3-5 words max)                    │
  └──────────────────────────────────────────────────────────────────┘

  Invocation flow:

  USER     ───►  "/git-branch-new PROJ-123"
                  │
                  ▼
  CLAUDE   ───►  matches against description in frontmatter
                  │
                  ▼
  RUNTIME  ───►  loads full SKILL.md into context

                  ├── executes Steps in order
                  ├── runs scripts/ when needed
                  ├── consults references/ for schemas or examples
                  └── reports structured output to the user
                  │

PHASE 1 — PLANNING · get task + create branch
  ┌──────────────────────────────────────────────────────────────────┐
  │ /git-branch-new PROJ-123                                         │
  │                                                                  │
  │ 1. reads task detail from tickets/backlog.md                     │
  │    - ID:           PROJ-123                                      │
  │    - title:        "DELETE /items/{id} with usage guard"         │
  │    - epic / description / cross-references                       │
  │                                                                  │
  │ 2. infers branch type from the ID + correct naming:              │
  │    PROJ-XXX        → PROJ-XXX-short-slug          (feature)      │
  │    PROJ-FIX-XXX    → PROJ-FIX-XXX-short-slug      (bug fix)      │
  │    PROJ-REL-XXX    → release/PROJ-REL-XXX         (release)      │
  │    hotfix/...      → hotfix/PROJ-FIX-XXX-slug     (urgent)       │
  │                                                                  │
  │ 3. detects the correct base branch                               │
  │    - feature              → develop                              │
  │    - release              → develop                              │
  │    - hotfix               → main                                 │
  │    - fix on active release → release/*                           │
  │                                                                  │
  │ 4. checks for ticket duplicates                                  │
  │    git fetch + grep ticket-id                                    │
  │    - exists        → direct switch (no new branch)               │
  │    - not exists    → proceeds                                    │
  │                                                                  │
  │ 5. auto-stash if uncommitted changes                             │
  │    git stash push -m "auto-stash before branch"                  │
  │                                                                  │
  │ 6. checkout base + pull + create branch                          │
  │    git checkout develop && git pull                              │
  │    git checkout -b PROJ-123-delete-item-endpoint                 │
  │                                                                  │
  │ → branch ready, base updated, ready to code                      │
  └──────────────────────────────────────────────────────────────────┘
PHASE 2 — DESIGN · think the feature before coding
  ┌──────────────────────────────────────────────────────────────────┐
  │ iterative conversation with Claude (not a skill)                 │
  │                                                                  │
  │ I iterate until the design closes:                               │
  │                                                                  │
  │   ME     ─►  context: reads CLAUDE.md + handlers/ + schema/      │
  │               how does this feature fit in the codebase?         │
  │               │                                                  │
  │               ▼                                                  │
  │   CLAUDE ─►  proposes a first approach + tradeoffs               │
  │               soft delete vs hard delete, status codes,          │
  │               edge cases it sees, files to touch...              │
  │               │                                                  │
  │               ▼                                                  │
  │   ME     ─►  question / refine / add constraints                 │
  │               "what if two concurrent requests?"                 │
  │               "what if the migration fails midway?"              │
  │               "I prefer soft delete for audit trail"             │
  │               │                                                  │
  │               ▼                                                  │
  │   CLAUDE ─►  adjusts the design + identifies new risks           │
  │               │                                                  │
  │               ▼                                                  │
  │   does the design close?                                         │
  │     NO  → question / refine again (loop)                         │
  │     YES → I have:                                                │
  │             - clear API contract                                 │
  │             - documented ADRs (decisions + why)                  │
  │             - list of files to touch                             │
  │             - tests to write before coding                       │
  │             - planned DB migration                               │
  │                                                                  │
  │ → only now I start coding (PHASE 3)                              │
  └──────────────────────────────────────────────────────────────────┘

  Without this iteration: you code → hit edge cases at review →
  repeat the PHASE 3 loop many more times. Thinking is cheap.
PHASE 3 — IMPLEMENTATION · Claude executes the plan
  ┌──────────────────────────────────────────────────────────────────┐
  │ Claude executes the plan from PHASE 2                            │
  │                                                                  │
  │ with the design plan in hand, Claude:                            │
  │                                                                  │
  │ 1. creates / modifies the files in the plan                      │
  │    - handlers/items_delete.go                                    │
  │    - store/items.go            (DeleteItem + guard)              │
  │    - migrations/014_items.sql  (forward + rollback)              │
  │                                                                  │
  │ 2. writes tests in parallel                                      │
  │    - unit:        store/items_test.go                            │
  │    - integration: handlers/items_delete_test.go                  │
  │    - covers edge cases identified in design                      │
  │                                                                  │
  │ 3. runs tests locally between changes                            │
  │    - immediate feedback without waiting for CI                   │
  │    - adjusts code if something side breaks                       │
  │                                                                  │
  │ 4. adds ADR references in code if applicable                     │
  │    eg: // ADR: soft delete via active flag — see PR #91          │
  │                                                                  │
  │ 5. I review what it did briefly:                                 │
  │    - the right files?                                            │
  │    - do the names make sense?                                    │
  │    - anything weird / hallucinated?                              │
  │                                                                  │
  │ anything off?                                                    │
  │   YES → I adjust / ask for changes → Claude iterates             │
  │   NO  → go to PHASE 4 (correction and validation)                │
  │                                                                  │
  │ → working code on the branch, ready to commit                    │
  └──────────────────────────────────────────────────────────────────┘

  The difference with "vibe coding": the plan already exists (PHASE 2),
  edge cases are already identified, tests know what to cover.
  Claude doesn't improvise — executes the plan + writes tests in parallel.
PHASE 4 — CORRECTION AND VALIDATION · iterative loop
  ┌──────────────────────────────────────────────────────────────────┐
  │ /git-ship-review   (orchestrator)                                │
  │                                                                  │
  │ STEP 1  - git-commit                                             │
  │   verifies branch (develop / main → ABORT)                       │
  │   generates conventional commit:                                 │
  │     feat(items): add DELETE endpoint                             │
  │     fix(parser): handle empty payload                            │
  │   stages + commits + pushes                                      │
  │                                                                  │
  │ STEP 2  - ci-test-report   (automatic validation)                │
  │   runs the project's full test suite:                            │
  │     go test ./...  → unit + integration tests                    │
  │     parses coverage per file                                     │
  │     detects broken / new / uncovered tests                       │
  │   did tests pass?                                                │
  │     NO → stop here, fix the code, back to STEP 1                 │
  │     YES → generates markdown table with coverage                 │
  │                                                                  │
  │ STEP 3  - git-pr-create                                          │
  │   detects base branch + syncs with remote                        │
  │   opens PR with Summary + Coverage table + Test plan             │
  │                                                                  │
  │ STEP 4  - git-pr-review   (semantic validation)                  │
  │   analyzes file by file + cross-impact:                          │
  │     ●  BLOCKER     cannot be merged                              │
  │     ▲  WARNING     review before                                 │
  │     ○  SUGGESTION  optional improvement                          │
  │                                                                  │
  │ any BLOCKERs?                                                    │
  │   YES → fix locally → back to STEP 1 (loop)                      │
  │   NO  → PR ready to merge to develop                             │
  └──────────────────────────────────────────────────────────────────┘

  Validation in 2 layers:
    ─ ci-test-report  → tests verify the code works
    ─ git-pr-review   → semantic analysis verifies it's well done
  The loop repeats until both layers return OK.
PHASE 5 — RELEASE · RC + validation
  ┌──────────────────────────────────────────────────────────────────┐
  │ /git-release-cut                                                 │
  │                                                                  │
  │ 1. creates branch release/v1.16.0 from develop                   │
  │                                                                  │
  │ 2. generates Release Candidate                                   │
  │      git tag v1.16.0-rc1                                         │
  │      go build → bin/v1.16.0-rc1/                                 │
  │      bin/current → rc1                                           │
  │                                                                  │
  │ 3. RC validation                                                 │
  │      - smoke tests on the binary                                 │
  │      - manual tests on critical endpoints                        │
  │      - DB migration verification                                 │
  │                                                                  │
  │ tests ok?                                                        │
  │   NO → fix in release branch → new RC (rc2, rc3...)              │
  │   YES → merge release → main                                     │
  │                                                                  │
  │ 4. final publication                                             │
  │      git tag v1.16.0  (no rc suffix)                             │
  │      go build → bin/v1.16.0/api/app                              │
  │      bin/current → v1.16.0  (atomic symlink)                     │
  │                                                                  │
  │ The deployment process always points to bin/current.             │
  │ The version is never hardcoded anywhere.                         │
  └──────────────────────────────────────────────────────────────────┘
PHASE 6 — BACKPORT
  ┌──────────────────────────────────────────────────────────────────┐
  │ /git-backport                                                    │
  │                                                                  │
  │ main (v1.16.0 already merged and tagged)                         │
  │      │                                                           │
  │      └── sync main → develop                                     │
  │          delegates to git-pr-create                              │
  │          → opens PR: backport v1.16.0 → develop                  │
  │                                                                  │
  │ → develop stays synced with main without losing commits          │
  └──────────────────────────────────────────────────────────────────┘
